# Story E06S19: Cache API L1 for Expert Pool

## User Story

**As a** platform operator,
**I want** expert pool reads to hit a per-datacenter Cache API layer before KV,
**so that** repeated reads within a 60-second window return in <1ms without any KV or DB roundtrip.

---

## Context

The expert pool is read on every prospect submit and match compute. Currently it goes KV (300s TTL) → DB fallback. Adding Cache API as L1 provides sub-millisecond reads for the majority of requests within a datacenter, reducing KV read costs and improving tail latency.

Cache API is per-datacenter (ephemeral), free, and requires no configuration beyond code changes.

---

## Acceptance Criteria

- [ ] AC1: `loadExpertPool()` checks `caches.default.match()` before KV `.get()`
- [ ] AC2: On Cache API hit → return directly, skip KV + DB
- [ ] AC3: On Cache API miss + KV hit → return from KV, write-back to Cache API
- [ ] AC4: On both miss → load from DB via Hyperdrive (postgres.js), write KV + Cache API
- [ ] AC5: Cache API TTL = 60 seconds (via `Cache-Control` header on cached Response)
- [ ] AC6: All tests pass

---

## Out of Scope

- D1 as intermediate cache layer (E06S23)
- Cache invalidation on expert registration (eventual consistency acceptable at 60s TTL)

## Dependencies

## Notes

Cache API uses Request/Response objects — the expert pool is stored as a JSON Response with `Cache-Control: max-age=60` header. Cache key: `https://cache.internal/expert-pool/{satellite_id}` (synthetic URL).
