# Story E06S11: Booking Engine — Availability, Create, Cancel, Reschedule

## User Story

**As a** prospect,
**I want** to see an expert's real-time availability and book a free discovery call,
**so that** I can connect with the right expert without any manual coordination or back-and-forth.

---

## Context

This story implements the full booking engine, replacing the booking capability previously delegated to Cal.com (E06S04, superseded — DEC-41). The engine is headless: it exposes API endpoints consumed by the satellite funnel widget (E06S07). All calendar operations are performed directly via Google Calendar API on behalf of the expert (OAuth tokens managed by E06S10). The booking flow is: availability query → hold slot → confirm (creates Google Calendar event + Meet link) → notify (via queue consumers, E06S06). Cancel and reschedule are also handled.

Anti double-booking is enforced at two levels: held slots excluded from availability responses, and a freebusy re-check at confirm time.

---

## Acceptance Criteria

- [ ] AC1: `GET /api/experts/:id/availability` queries Google Calendar Freebusy API (`POST /calendar/v3/freeBusy`) for the expert's primary calendar over the next 14 days (configurable via `experts.availability_rules.booking_window_days`), computes free 20-minute slots by applying: working hours (`availability_rules.working_hours`), buffer time (`availability_rules.buffer_minutes`, default 15), minimum notice (`availability_rules.min_notice_hours`, default 4), existing `bookings` with `status IN ('held','confirmed')` — returns `{ slots: [{ start_at, end_at }] }` in UTC; `?tz=Europe/Paris` query param is accepted and echoed back in response metadata but slots remain in UTC
- [ ] AC2: `experts.availability_rules` JSONB defaults (applied when null): `{ working_hours: { mon: ["09:00","18:00"], tue: ["09:00","18:00"], wed: ["09:00","18:00"], thu: ["09:00","18:00"], fri: ["09:00","18:00"] }, buffer_minutes: 15, min_notice_hours: 4, booking_window_days: 14 }` — weekends excluded by default
- [ ] AC3: `POST /api/bookings/hold` accepts `{ expert_id, start_at, end_at, prospect_id }`, verifies the slot is not already held or confirmed (DB check), creates a `bookings` row with `status: 'held'` and `held_until: now() + 10 minutes` — returns `{ booking_id, held_until }` — returns `409` if slot already taken
- [ ] AC4: `POST /api/bookings/:id/confirm` accepts `{ prospect_name, prospect_email, description? }`: (1) re-checks slot availability via Google Freebusy (race condition guard), (2) creates Google Calendar event on expert's primary calendar via `events.insert` with `conferenceDataVersion=1` (fresh Google Meet link — never reused), attendees: expert + prospect, (3) updates `bookings` row: `status: 'confirmed'`, `gcal_event_id`, `meeting_url`, `prospect_name`, `prospect_email`, `description`, `confirmed_at` — returns `{ booking_id, meeting_url, start_at, end_at, expert_gcal_event_id }`
- [ ] AC5: On `confirm` success, pushes two queue messages: `{ type: "booking.created", booking_id, expert_id, prospect_id }` → `callibrate-core-queue-lead-billing-{env}`; `{ type: "booking.confirmed", booking_id }` → `callibrate-core-queue-email-notifications-{env}`
- [ ] AC6: Anti double-booking: if freebusy re-check at confirm time reveals the slot is taken (race condition), returns `409 Conflict` with `{ error: "slot_taken" }` — no Google Calendar event created, no queue messages pushed, `bookings` row deleted
- [ ] AC7: `DELETE /api/bookings/:id` (cancellation by prospect or expert): calls Google Calendar API to delete the event (`events.delete`, `gcal_event_id`), updates `bookings.status = 'cancelled'`, pushes `{ type: "booking.cancelled", booking_id }` → `email-notifications` queue — returns `200`; if `gcal_event_id` is null (booking not yet confirmed), skips Google delete and only updates DB
- [ ] AC8: `POST /api/bookings/:id/reschedule` accepts `{ new_start_at, new_end_at }`: checks new slot availability (freebusy + DB), updates Google Calendar event times (`events.patch`), updates `bookings` row with new times, pushes `{ type: "booking.rescheduled", booking_id }` → `email-notifications` queue — returns `{ booking_id, meeting_url, new_start_at, new_end_at }`; returns `409` if new slot taken
- [ ] AC9: Expired holds cleanup: Cloudflare Workers Cron Trigger (every 5 minutes) deletes `bookings` rows where `status = 'held' AND held_until < now()` — making those slots available again in availability responses
- [ ] AC10: Google Calendar API error handling: `429` → exponential backoff, up to 3 retries (delays: 1s, 4s, 16s); `401` → invoke `refreshGcalToken` (E06S10) then retry once; `404` on event delete (already deleted externally) → treat as success, update DB; all other errors → return `502` with `{ error: "gcal_error", status, message }`
- [ ] AC11: Schema migration adds: `experts.availability_rules jsonb`, `bookings.held_until timestamptz`, `bookings.gcal_event_id text`, `bookings.meeting_url text`, `bookings.prospect_name text`, `bookings.prospect_email text`, `bookings.description text`, `bookings.confirmed_at timestamptz`
- [ ] AC12: At least 5 vitest tests: (1) availability computation with mock freebusy (buffer, working hours, held slots excluded); (2) hold creation — happy path + 409 on conflict; (3) confirm with mock Google Calendar insert — correct event structure, conferenceDataVersion=1; (4) cancel — mock event delete; (5) anti double-booking — 409 when freebusy re-check detects conflict at confirm time

## Notes

- Google Meet link is generated per booking via `conferenceData.createRequest` — **never reuse** conference data between bookings (deep research: reuse causes access issues and exposes meeting details to unintended participants).
- The hold TTL (10 minutes) is a product decision: long enough for the prospect to fill in their name/email, short enough to not block slots for too long. Adjustable post-MVP.
- `availability_rules` JSONB is set during expert onboarding (E02S02) — this story only reads it. Default values are applied if null.
- The Cron Trigger for expired holds must be declared in `wrangler.toml` under `[triggers] crons`.
- `bookings.prospect_id` (existing column from E06S02) is set from the satellite session token — not from a logged-in user account (prospects are anonymous).
- Google Calendar API quota: 1,000,000 requests/day per project (free tier). At MVP scale, not a concern. Rate limit is 10 requests/second per user — handle with backoff per AC10.
