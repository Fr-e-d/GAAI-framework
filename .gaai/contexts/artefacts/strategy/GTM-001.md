# GTM-001: Callibrate — Plan de Go-to-Market

> Principe directeur : un gate à la fois. La phase suivante ne démarre
> que lorsque le gate précédent est PASS. Pas d'exception.

---

## Vue d'ensemble

```
Phase 0 — Signal          J0–J7    ← MAINTENANT (parallèle avec Phase 1)
Phase 1 — Fondations      J0–J21   ← MAINTENANT (parallèle avec Phase 0)
Phase 2 — Premier cohort  J21–J45  ← débloquée par Gate 0 + Gate 1
Phase 3 — Monétisation    J45–J90  ← débloquée par Gate 2
```

---

## Phase 0 — Signal de marché (J0–J7)

**Objectif :** Prouver que les deux côtés du marché veulent ce produit
avant d'investir une heure de build supplémentaire.

**Actions :**
- J1–J3 : Reddit + Slack/Discord — écoute, affinage du langage de promesse (E01S01)
- J3–J7 : Campagne Hand Raiser experts — LinkedIn + X + Facebook (E01S02)
- J3–J7 : Campagne Hand Raiser prospects — LinkedIn + X + Facebook (E01S03)

**Gate 0 — SMART**

| Critère | Mesure | Seuil |
|---|---|---|
| Experts main levée | Commentaires publics qualifiés | ≥ 10 |
| Prospects main levée | Commentaires publics qualifiés | ≥ 15 |
| Deadline | J7 | Hard |

```
PASS → Phase 2 débloquée. Promesse validée et verrouillée.
FAIL → Réécrire la promesse (1 retry max, 7 jours).
       Si 2ème FAIL → pivot ou stop. Ne pas builder sur du sable.
```

## Phase 2 — Premier cohort + première boucle réelle (J21–J45)

**Objectif :** 10 experts réels onboardés. Premiers prospects réels
qualifiés. Premiers appels bookés. Signal de qualité produit.

**Prérequis :** Gate 0 PASS + Gate 1 PASS

**Actions :**
- Contacter les hand raisers experts de Phase 0 (founder-led)
- Onboarding manuel des 10 premiers experts (profil + Cal.com)
- Lancer le premier satellite site (vertical le plus simple)
- Contacter les hand raisers prospects de Phase 0
- Premiers quiz soumis → premiers matchings → premiers bookings
- Collecter les premiers post-call surveys (prospects + experts)

**Gate 2 — SMART**

| Critère | Mesure | Seuil |
|---|---|---|
| Experts vérifiés | Profils complets + calendrier connecté | ≥ 10 |
| Prospects soumis | Quiz completions + extraction AI | ≥ 20 |
| Appels bookés | Cal.com bookings confirmés | ≥ 5 |
| Score appel moyen | Post-call survey prospects | ≥ 4/5 |
| Conversion | Prospects → booking | ≥ 35% |
| Deadline | J45 | Hard |

```
PASS → Phase 3 débloquée. Le produit délivre de la valeur réelle.
FAIL → Analyser le goulot :
  → Peu de soumissions ? → Problème acquisition prospect / funnel
  → Peu de matchings ? → Problème engine / profils experts
  → Peu de bookings ? → Problème reveal / booking UX
  → Score < 4/5 ? → Problème qualité experts / brief prospect
  Ajuster le point de friction. Ne pas tout changer en même temps.
```

## Phase 3 — Monétisation + feedback loop (J45–J90)

**Objectif :** Premier revenu réel. Experts paient pour leurs leads.
Feedback loop tri-directionnel actif. Plateforme auto-améliorante.

**Prérequis :** Gate 2 PASS

**Actions :**
- Activer E05 : pay-per-call checkout (Lemon Squeezy)
- Activer lead evaluations (expert note le lead J+48)
- Activer post-projet satisfaction (prospect J+45)
- Activer composite score computation
- Lancer le dashboard expert avec scores visibles
- Itérer sur les matching weights avec les données de conversion

**Gate 3 — SMART**

| Critère | Mesure | Seuil |
|---|---|---|
| Calls payés | Transactions Lemon Squeezy | ≥ 10 |
| GMV | Revenu total transactions | ≥ 1 500€ |
| Expert satisfaction | Score dashboard NPS experts | ≥ 7/10 |
| Experts re-payants | Experts ayant payé ≥ 2 leads | ≥ 3 |
| Lead eval rate | % experts ayant noté leurs leads | ≥ 60% |
| Deadline | J90 | Hard |

```
PASS → Décision : lever du seed / continuer bootstrapped / expansion verticale
FAIL → Analyser :
  → Experts ne paient pas ? → Pricing, valeur perçue, qualité lead
  → GMV faible ? → Volume de calls insuffisant → retour Phase 2
  → Eval rate < 60% ? → Onboarding / incentive à revoir
```

---

## Règles d'exécution

1. **Un seul gate à la fois.** Ne pas commencer Phase 3 si Gate 2 est
   incertain. Les gates sont binaires : PASS ou FAIL.

2. **Phase 0 et Phase 1 sont parallèles.** Le build ne dépend pas
   du signal — mais Phase 2 dépend des deux.

3. **Ne pas over-engineer entre les gates.** Chaque phase livre
   le minimum nécessaire pour valider le gate suivant.

4. **Chaque FAIL est une décision, pas un échec.** Il indique
   quoi ajuster. Un seul levier à la fois.

5. **La mémoire GAAI est mise à jour à chaque gate.** Les décisions
   prises en réponse à un FAIL sont capturées comme décisions validées.
